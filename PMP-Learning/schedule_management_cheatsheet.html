2. Rolling Wave Planning

üìä Iterative Planning

Rolling wave planning is an iterative, progressive detailing technique. Work far in the future is planned at a high level; work near-term is planned in detail.

How It Works
  • High-level planning for distant work.
  • Detailed planning when work is near execution.
  • Revisited regularly ‚Äî plans ‚Äúroll forward.‚Äù
  • Reduces waste vs planning everything upfront.
Why It‚Äôs Useful
  • Supports uncertainty and evolving requirements.
  • Strong fit with Agile ‚Äì detailed sprint-level planning.
  • Works with decomposition (WBS ‚Üí work packages).
üß≠ Exam hint: When the question says ‚Äúplan in detail only for near-term work,‚Äù the answer is always Rolling Wave Planning.

3. Leads & Lags

üß∞ Schedule Adjustment

Leads and lags adjust the timing between activities to optimise the schedule.

Lead (Start Earlier)
  • Brings the next activity forward.
  • Example: Start editing photos before shooting is finished.
  • Reduces total project duration.
Lag (Delay Start)
  • Introduces a waiting period.
  • Example: Wait 5 days for concrete to cure before framing.
  • Represents necessary delay.
üß≠ Remember: Leads shorten the schedule. Lags extend it (when needed).

4. Precedence Diagramming Method (PDM)

üìä Dependencies

PDM is used to sequence activities and show logical relationships between them. It is central to building the network diagram and identifying the critical path.

Four Relationship Types
  • Finish to Start (FS): Most common ‚Äî Task B starts after A finishes.
  • Start to Start (SS): B starts after A starts.
  • Finish to Finish (FF): B finishes after A finishes.
  • Start to Finish (SF): B finishes after A starts (least common).
Examples
  • FS: Build PC ‚Üí Install software.
  • SS: Start pouring concrete ‚Üí Start leveling.
  • FF: Finish writing ‚Üí Finish editing.
  • SF: Start new AP system ‚Üí Shutdown old system.
üß≠ Exam hint: FS is by far the most common and default dependency.

5. Estimating Techniques

üìä Duration Estimation

Four core duration estimating methods you must know well.

Parametric
  • Uses measurable unit rate.
  • Example: GHS 55 per sq ft, 8 hours per risk assessment.
  • Good accuracy when data is strong.
Analogous
  • Based on similar past project.
  • Fast and cheap.
  • Low accuracy.
Bottom-Up
  • Estimate work packages individually.
  • Sum them to form total estimate.
  • Most accurate, most time-consuming.
Three-Point
  • Optimistic, Most Likely, Pessimistic.
  • Simple average of the three.
  • Helps when uncertainty exists.

6. Beta / PERT Estimating

üß∞ Weighted Estimation

PERT is a variant of three-point estimating that weights the most likely estimate four times.

Formula

(Optimistic + 4 √ó Most Likely + Pessimistic) √∑ 6

7. Resource Leveling vs Resource Smoothing

üìä Resource Optimization

Resource optimization fixes overloaded resources or schedule risks by adjusting activity timing.

Resource Leveling
  • Adjusts schedule based on resource limits.
  • Fixes over-allocation.
  • May extend the project end date.
  • Example: Sue cannot do two tasks at once ‚Üí move tasks to new days.
Resource Smoothing
  • Delays activities only within float.
  • Does NOT change project end date.
  • Uses free and total float.
  • Activities without float cannot be smoothed.
üß≠ Key difference: Leveling may delay the project. Smoothing cannot delay the project.

8. Critical Path Method

üìä Scheduling Core

The critical path is the longest sequence of activities with zero float and determines the shortest possible overall project duration.

How It Works
  • Use PDM to define activity dependencies.
  • Perform a forward pass ‚Üí calculate early start/finish.
  • Perform a backward pass ‚Üí calculate late start/finish.
  • Float = Late Start ‚àí Early Start.
  • Critical path = activities with zero float.
üß≠ Memory tip: Longest path = shortest schedule. Critical path has zero float.

9. Schedule Compression Techniques

üß∞ Acceleration

Compression reduces schedule duration without changing project scope.

Crashing
  • Add resources or money.
  • Often increases cost significantly.
  • Used only on tasks on the critical path.
  • Can increase risk (overtime, fatigue, errors).
Fast Tracking
  • Overlap sequential tasks.
  • Uses the same resources ‚Äî no added cost.
  • Works only when activities can be parallelised.
  • Often increases risk (rework, errors).
üß≠ Key rule: Crash = add resources. Fast-track = overlap tasks. Both apply only to activities on the critical path.
