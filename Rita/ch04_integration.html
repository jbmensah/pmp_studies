üìö Integration Management Overview

Integration is the process of balancing all project activities with each other while acting as a project ambassador for all stakeholders.

Key Concepts

  • The PM's primary role is performing integration management.
  • It requires the PM to be an accomplished, innovative thinker, collaborative leader, and possess business savvy.
  • Integration ensures project needs align with related portfolios and programs within the larger organization.
  • Project management activities are interdependent (e.g., budget is affected by time, resources, and risk costs).
üí° Trick of the Trade: Integration Ambassador
Think of integration as balancing all project activities (constraints) while simultaneously being the project ambassador to stakeholders (team, customers, management, regulators).

1. Develop Project Charter üìú

Process Group: Initiating (Task 9: Initiate project planning activities starts here)

Purpose: Formal authorization for the project to continue. It assesses feasibility within constraints at a high level and grants the Project Manager authority to apply organizational resources.

Key Inputs (I)

  • Agreements / Contracts
  • Business Documents (Business Case / Benefits Management Plan)

Key Outputs (O)

  • Project Charter (Crucial artifact, signed off by the sponsor)
  • Assumption Log (Starts here; includes initial assumptions and constraints)
üìú Memory Hook: Authority Granted, Assumptions Logged.
Agile Note: Tailored charter documents may articulate more uncertainty about requirements. The Product Owner may have authority over change management and prioritizing requirements instead of a formal CCB.

2. Develop Project Management Plan üó∫Ô∏è

Process Group: Planning

Purpose: To integrate all individual management plans (scope, schedule, cost, quality, risk, etc.) and baselines into a cohesive, centralized document that describes how the project will be executed and controlled.

PMP Core Components

The PMP includes management plans for 10 knowledge areas, plus the Change, Configuration, and Requirements Management Plans.

Performance Measurement Baseline (PMB): The integrated baselines:

  • Scope Baseline: Project Scope Statement, WBS, WBS Dictionary.
  • Schedule Baseline: Agreed-upon schedule with start/stop dates/milestones.
  • Cost Baseline: Time-phased cost budget (spending plan).

Key Outputs (O)

  • Project Management Plan (PMP) (Must be bought into, approved, realistic, and formal)
  • Baselines are included in the PMP.
üó∫Ô∏è Memory Hook: The Blueprint, Baselines Defined.
‚ö†Ô∏è Crucial Reminder: The PMP is a set of plans and baselines, not just a schedule. Assume management plans (Change, Config, Requirements) have been created. If a problem arises, the PM should consult the corresponding management plan first.
Agile Note: Initial plans may be deliberately light and progressively elaborated (release / iteration planning). Scope, schedule, and risk plans are often incorporated into the product backlog and release road maps.

3. Direct and Manage Project Work üõ†Ô∏è

Process Group: Executing

Purpose: To perform the work defined in the PMP, producing project deliverables, and integrating all executing work into one coordinated effort. This includes managing work, keeping people engaged, and facilitating issue resolution.

Key Inputs (I)

  • Project Management Plan (PMP)
  • Approved Change Requests (Work resulting from approved changes is completed here)

Key Outputs (O)

  • Deliverables (Products / services / results produced)
  • Work Performance Data (WPD) (Raw measurements, gathered here)
  • Issue Log (Created and used here)
  • Change Requests (Identified here)
üõ†Ô∏è Memory Hook: Executing the Plan, Generating Data (WPD).
Plan-Driven Tool: A Work Authorization System (WAS) may be used to ensure work is only started when formally authorized, preventing conflicts (e.g., plumber / electrician conflict).

4. Manage Project Knowledge üß†

Process Group: Executing

Purpose: To use existing organizational knowledge and create/share new knowledge (lessons learned) to benefit the current project and make the organization better for future projects.

Knowledge Types

  • Explicit Knowledge: Fact-based, easily communicated (e.g., documentation, processes, traditional lessons learned).
  • Tacit Knowledge: Not easily documented (e.g., emotions, experience, ability; learned through job shadowing / apprenticeship). The PM needs to foster a trusting environment for sharing both.

Key Inputs (I)

  • Project Artifacts (e.g., PMP, Documents, Deliverables)

Key Outputs (O)

  • Lessons Learned Register Updates (Main output; describes what was right / wrong / what to do differently)
  • Organizational Process Assets Updates (Knowledge is documented in OPAs for future projects)
üß† Memory Hook: Lessons Learned Shared (Explicit & Tacit).

Osmotic Communication: Informal knowledge sharing facilitated by physical proximity among team members.

5. Monitor and Control Project Work üî≠

Process Group: Monitoring & Controlling

Purpose: To observe, track, review, and report overall project performance. It aggregates work data to evaluate impacts on plans and baselines, enabling the PM to take corrective actions.

Key Activities

This process encourages a holistic view of performance (e.g., scope met, but cost excessive).

  • Analyzing and tracking risks.
  • Performing data analysis (Trend, Variance, Earned Value, Root Cause).
  • Reviewing change effectiveness.

Key Inputs (I)

  • Work Performance Information (WPI) (Processed data, comes from other M&C processes)

Key Outputs (O)

  • Work Performance Reports (WPR) (Performance metrics communicated to stakeholders)
  • Change Requests (Generated when variances or issues are discovered)
üî≠ Memory Hook: Watching Performance, Aggregating WPI.
WPD vs WPI vs WPR:
  1. WPD (Data) is raw, gathered in Direct & Manage Project Work (Executing).
  2. WPD is processed into WPI (Information) in Monitoring processes.
  3. WPI is compiled into WPR (Reports) in Monitor & Control Project Work (Integration M&C).

6. Perform Integrated Change Control ‚öñÔ∏è

Process Group: Monitoring & Controlling (Task 10: Manage project changes)

Purpose: To review all change requests and either approve, reject, or defer them. The key focus is assessing the impact of a change on ALL project constraints (scope, schedule, cost, risk, quality, etc.) before implementation.

Categories of Change Requests (CRs)

  • Corrective Action: Brings expected future performance in line with the PMP (deals with ACTUAL deviations).
  • Preventive Action: Deals with ANTICIPATED deviations (e.g., adjusting the project to prevent a problem from recurring).
  • Defect Repair: Rework required when a component does not meet requirements.
1. Evaluate Impact (on ALL constraints)
2. Identify Options (e.g., crashing, cutting scope)
3. Get Approval (CCB / Sponsor)
4. Implement Change (in Direct & Manage Work)

Key Inputs (I)

  • Project Management Plan (PMP)
  • Change Requests (Generated from many processes)

Key Outputs (O)

  • Approved Change Requests (Input to Direct & Manage Project Work)
  • Change Log (Documents the status and reason for approval/rejection)
  • PMP Updates & Project Document Updates (If baselines or plans change)
‚öñÔ∏è Memory Hook: Balance Constraints, Approve / Reject / Defer.
‚ö†Ô∏è When is Integrated Change Control needed? It is required when changes affect the approved Project Charter, PMP, or Baselines. Changes to project documents (like the Issue Log) do not usually require formal CRs or CCB approval.
CCB (Change Control Board): Assume plan-driven projects have a formal CCB (Steering Committee) responsible for review. Changes to contracts, baselines, or the charter typically require CCB/Sponsor approval.
Deviation from Baselines: If a project deviates significantly, the PM should often review the Risk Management process for incompleteness.

7. Close Project or Phase ‚úÖ

Process Group: Closing (ECO Task 17: Plan and manage project phase closure or transitions)

Purpose: To finalize all activities across tasks and processes to formally conclude the project or phase, ensure final requirements are met, and transition the final product.

Mandatory Closing Steps

Closing must be done, even if the project is terminated early.

  • Obtain Final Formal Acceptance of the product/project from the customer.
  • Complete financial closure and final procurement closure.
  • Issue final performance reporting.
  • Index and archive all project records.
  • Gather final lessons learned and update the knowledge base (OPAs).

Key Inputs (I)

  • Project Management Plan (PMP)
  • Accepted Deliverables (from Validate Scope)

Key Outputs (O)

  • Final Product, Service, or Result Transition
  • Organizational Process Assets Updates (Final archiving and lessons learned)
  • Final Report
‚úÖ Memory Hook: Formal Acceptance, Final Archive.
Closure vs. Validation: Close Project or Phase obtains formal acceptance of the entire project/phase from the customer. Validate Scope (M&C process) obtains formal acceptance for interim deliverables. The project needs both.
